## 为什么需要拆开这些层

同一段业务代码，仍然使用 OpenAI SDK、相同的 `messages` 和 `tools` 字段，只是切换了 `base_url` 与模型，system 指令、工具调用、JSON 稳定性、流式事件和长上下文表现就可能全部变样。

这不是某个供应商的兼容性缺陷。API 表面之下还有一整套未被 JSON 统一的协议与运行边界。需要回答的是：

1. 请求 JSON 进入模型之前，到底经历了什么？
2. 工具为什么不是“多传一个 schema”这么简单？
3. KV Cache、Prefix Cache 与调度为何会成为长上下文和多租户的约束？
4. 哪些结论来自公开资料，哪些只能作为工程判断？

---

## 一页结论

**OpenAI-like API 只意味着接入层大致兼容，不意味着模型行为等价于 GPT。**

一次调用的实际行为由两条相互关联、但不应混为先后顺序的路径共同决定：

```text
离线：训练与对齐
数据 → tokenizer / special tokens → pretraining / SFT / RLHF / RLAIF → 模型权重

在线：一次请求
messages / tools JSON → chat template → token IDs
                          ↓
          serving runtime（batching / KV cache / prefix cache / scheduling / constraints）
                          ↓
                    使用模型权重推理 → structured output / tool call / streaming → response
```

训练与对齐塑造模型参数；Serving Runtime 在在线阶段组织模型推理。二者共同影响最终行为，但不是“先跑模型、再跑运行时”的串行步骤。

页面顶部的图只画在线请求链路：API 收到的是结构化对象，模板与 tokenizer 将其变成模型可消费的 token；Serving Runtime 则在内部调度模型计算、KV Cache 和请求队列，随后产生响应。

因此，同样使用 OpenAI SDK、同样调用 `/v1/chat/completions` 或 `/v1/responses` 的两个服务，仍可能在以下方面明显不同：

- system / developer 指令的优先级与转写策略；
- 工具是否调用、调用哪个工具、参数是否稳定；
- JSON / schema 输出稳定性；
- thinking / reasoning 输出方式；
- 长上下文质量、截断和停止行为；
- 首 token 延迟（TTFT）、token 间延迟（ITL）；
- Prefix Cache 命中与多租户公平性；
- 安全边界、拒答方式与默认产品策略。

`developer` 在新 API 里已经不算常见写法，但它并没有消失。OpenAI Chat Completions 参考中仍有 `ChatCompletionDeveloperMessageParam`，并说明 o1 及更新模型用 `developer` message 替代旧的 `system` message 用法；Responses API 的 input role 也包含 `developer`，且 `instructions` 被描述为插入模型上下文的 system 或 developer message。[S19][S20]

但这仍然只是 API 对象层的 role。进入具体开源 serving 时，它可能被原样交给模型模板，也可能被运行时转写。vLLM 当前会检测 chat template 是否包含 `developer` role；模板不支持时，会把 `developer` 转成 `system` 再渲染。[S21]

**可操作原则：**把“OpenAI-compatible”视为**传输与对象结构兼容**，而不是**行为同构合同**。

## 0. 框架地图：vLLM 在哪里

LLM 推理与服务框架大致可以分成几条路线。它们不是完全可互换的“同类产品”，而是分别优化不同硬件、模型、并发与部署边界。

| 路线 | 代表项目 | 更适合的场景 | 说明 |
|---|---|---|---|
| 通用 GPU 高并发 Serving | [vLLM](https://github.com/vllm-project/vllm)、[SGLang](https://github.com/sgl-project/sglang) | 云端推理、长上下文、Agent、OpenAI-like API | vLLM / SGLang 都重视连续批处理、KV 管理与前缀复用。 |
| NVIDIA 深度优化 | [TensorRT-LLM](https://github.com/NVIDIA/TensorRT-LLM) | NVIDIA 生产集群、专门推理平台团队 | 依赖 NVIDIA 推理栈，通常更重工程调优。 |
| 中文模型 / Qwen / 多模态部署 | [LMDeploy](https://github.com/InternLM/lmdeploy) | Qwen、InternLM、国产部署、量化与 VLM | 与相关模型生态结合较深。 |
| 本地与边缘设备 | [llama.cpp](https://github.com/ggml-org/llama.cpp) | Mac、CPU、GGUF、个人设备、边缘推理 | 不以数据中心高并发 Serving 为主赛道。 |
| 跨端编译部署 | [MLC LLM](https://github.com/mlc-ai/mlc-llm) | Web、Android、iOS、异构硬件 | 更偏编译与跨端运行。 |
| 训练生态延伸 | [DeepSpeed](https://github.com/microsoft/DeepSpeed) | 已深度使用 PyTorch / DeepSpeed 的团队 | 更适合结合训练与推理基础设施理解。 |

这些项目的许可证、具体支持模型、API 能力和版本状态会变化。做选型时应以各项目当前仓库和文档为准，不要把某次对比结论雕成碑文。

## 1. 为什么 OpenAI-like API 不等于 GPT 行为

### 1.1 API 兼容解决了什么

一个 OpenAI-compatible 服务通常承诺以下一部分能力：

```python
from openai import OpenAI

client = OpenAI(
    base_url="https://provider.example/v1"
)
```

应用可以沿用类似的：

- endpoint；
- HTTP / JSON 字段；
- streaming 事件形态；
- `messages`、`tools`、`response_format` 等对象；
- 基础错误与用量字段。

这很好，因为业务接入成本会下降。只是它并不保证同一输入产生同一模型行为。

### 1.2 真正导致行为差异的层

| 层 | 影响的内容 |
|---|---|
| 预训练数据 | 模型见过哪些语言、代码、知识模式和任务分布。 |
| Tokenizer | 文本如何切分，特殊 token 如何编码，上下文成本如何计算。 |
| SFT | 模型学习如何作为助手回答、如何写 JSON、何时调用工具。 |
| RLHF / RLAIF | 模型偏好什么风格、何时拒答、如何权衡帮助性与安全性。 |
| Chat Template | `messages` 与 `tools` 到底被串成怎样的 token 序列。 |
| Tool parser / constraints | 如何识别工具调用，是否强制 JSON Schema 或 grammar。 |
| Serving runtime | batching、量化、KV cache、调度、超时、截断、流式策略。 |
| 产品策略 | 隐藏 system prompt、模型路由、安全过滤、默认参数、工具沙箱。 |

### 1.3 兼容到底兼容到哪一层

“OpenAI-compatible”至少可能指四件不同的事，不能混为一句承诺：

| 层级 | 含义 | 能否推出 GPT 等价行为 |
|---|---|---|
| SDK 兼容 | 客户端改 `base_url` 后可以调用。 | 不能。 |
| HTTP / JSON 兼容 | endpoint、字段、基础错误和部分流式形态相近。 | 不能。 |
| 语义兼容 | role 优先级、工具选择、结构化输出、停止与错误行为也一致。 | 很少能完整做到。 |
| 模型行为等价 | 质量、风格、推理、拒答和代码能力都接近同一模型。 | 不能由 API 兼容推出。 |

这张表不是定义游戏，而是测试边界。跨供应商接入时，不能只测“请求是否 200”；还要分别测指令冲突、工具调用、schema、流式事件、超长输入、重试和拒答。

### 1.4 关于 GPT 与 Claude 的底层部署框架

**未公开 / 不应假定：**OpenAI 与 Anthropic 没有在其面向开发者的公开文档中完整披露 GPT / Claude 的核心生产推理栈，也没有公开承诺“GPT API 就是某一个开源框架原样运行”。

可以确认的是：它们都提供模型 API、工具与 Agent 能力；但把“OpenAI / Anthropic 使用了某个开源框架”当作确定事实是不严谨的。

工程上更安全的结论是：

> 闭源大模型服务通常应被看作“自研基础设施 + 可能的开源组件 + 硬件特定优化 + 未公开控制面”的组合；除非厂商明确披露，否则不要假设其等同于某一开源 Serving 框架。

## 2. 训练与对齐：模型为什么会“像这样说话”

### 2.1 预训练：决定模型“会不会”

预训练通常让模型在海量 token 序列上执行 next-token prediction：根据前文预测下一个 token。

```text
输入：KV Cache 可以避免重复计算……
目标：预测下一个 token 的概率分布
```

预训练会形成语言、代码、数学、知识与模式识别能力的基础。它更接近“模型会不会某类任务”的能力地基。

OpenAI 在 InstructGPT 的公开说明中解释：基础语言模型通过互联网文本上的 next-word prediction 学习，而这种目标本身并不等于“安全、真实、按用户意图完成任务”。

来源：[OpenAI, Aligning language models to follow instructions](https://openai.com/index/instruction-following/)

### 2.2 Tokenizer：模型并不直接读文本

模型最终处理的是 token IDs，而不是自然语言字符。

```text
文本
→ tokenizer
→ token 序列
→ token IDs
→ embeddings
→ Transformer
```

Tokenizer 常包括：

- 文本归一化规则；
- BPE / SentencePiece 等切分模型；
- token 到 ID 的映射；
- BOS、EOS、role marker、tool marker 等特殊 token。

在主流 LLM 中，tokenizer 常先训练并固定；模型随后学习对应 token ID 的 embedding 与后续参数。也就是说，embedding 会更新，但 token-ID 对照关系通常不会在预训练中随意重写。

SentencePiece 项目将自身描述为用于神经文本生成的无监督 tokenizer；其典型工作流是先从语料训练 tokenizer / vocabulary，再供神经网络训练使用。

来源：[SentencePiece](https://github.com/google/sentencepiece)

#### Tokenizer 会不会限制模型能力上限？

**工程结论：会强烈影响有效能力，但很难简单说它单独决定“智能上限”。**

Tokenizer 会显著影响：

- 同一文本所需 token 数；
- 上下文窗口可容纳的真实字符量；
- 推理成本与 KV Cache 占用；
- 多语言公平性；
- 代码、罕见符号和领域文本的切分效率；
- 特殊控制 token 是否正确进入模型上下文。

例如，若某语言在同一内容下被切得更碎，那么它会消耗更多上下文和 KV Cache。模型不一定“完全看不懂”，但成本和长文本能力会变差。

### 2.3 SFT：决定模型“如何回答”

SFT（Supervised Fine-Tuning）使用“输入 → 理想回答”的示范数据。

```text
用户：解释 KV Cache。
助手：KV Cache 是 Transformer 推理时……
```

典型目标是提高目标答案每一个 token 的概率：

```text
L_SFT = -Σ log πθ(y*t | x, y*<t)
```

直觉：

```text
模型输出 token
→ 与标准 token 对比
→ 计算 loss
→ 反向传播
→ 提高标准 token 的概率
```

因此 SFT 的反馈很直接：

> “在这个位置，应该更倾向生成哪个 token。”

SFT 会塑造角色遵循、格式、代码风格、工具调用格式、拒答方式和 JSON 输出习惯。

### 2.4 RLHF：从人类偏好到主模型参数

RLHF（Reinforcement Learning from Human Feedback）常使用回答偏好对，而非仅使用标准答案。

```text
同一问题 x：
回答 A：更准确、清楚、符合要求
回答 B：较差

人类标注：A > B
```

#### Reward Model 做什么

Reward Model 输入“问题 + 完整回答”，输出标量 reward：

```text
rφ(x, y)
```

训练目标是让它认为人类偏好的回答分数更高：

```text
L_RM = -log σ(rφ(x, yw) - rφ(x, yl))
```

其中：

- `yw`：胜出的回答；
- `yl`：较差回答；
- `σ`：sigmoid。

Reward Model 更像“学习了人类偏好判断的裁判”，不是最终直接回答用户的主模型。

#### Reward 如何作用于主模型参数

典型流程：

```text
主模型 πθ 生成回答 y
        ↓
Reward Model 给分 rφ(x, y)
        ↓
优化算法根据 reward 更新主模型 θ
```

公开的 InstructGPT 流程是：先收集人类示范训练 SFT 模型，再收集人类偏好排序训练 Reward Model，最后使用 PPO 根据 Reward Model 优化主模型策略。

来源：[OpenAI, Aligning language models to follow instructions](https://openai.com/index/instruction-following/)

概念上目标可写为：

```text
最大化 E[rφ(x, y)] - β · KL(πθ || πref)
```

第一项鼓励高奖励回答；KL 项限制模型偏离参考模型过远，避免模型为了“骗高分”而过度漂移。

#### SFT 与 RLHF 的区别

| 项目 | SFT | RLHF |
|---|---|---|
| 人类信号 | 标准答案 | 回答偏好排序 |
| 监督粒度 | token 级 | 整体回答级 |
| 主要目标 | 拟合示范 | 最大化偏好 reward |
| 主模型梯度来源 | 标准 token 的交叉熵 | reward、策略梯度与 KL 约束 |
| 常见风险 | 过度拟合示范形式 | 讨好、奖励黑客、过度拒答或过度格式化 |

底层都仍然是“前向传播 → loss → backprop → optimizer → 参数更新”。差别在于：SFT 直接告诉模型“写什么”；RLHF 更像告诉模型“这整条回答总体更受偏好”。

### 2.5 RLAIF 与 Constitutional AI

RLAIF（Reinforcement Learning from AI Feedback）与 RLHF 思路类似，但部分偏好判断由 AI 生成或辅助。

Anthropic 的 Constitutional AI 是公开代表路线：先提供一组原则（constitution），让模型生成、批评和修订回答，并以 AI 反馈参与后续对齐。

来源：[Anthropic, Constitutional AI: Harmlessness from AI Feedback](https://www.anthropic.com/research/constitutional-ai-harmlessness-from-ai-feedback)

**核心记忆：**

```text
预训练：决定“会不会”。
SFT：决定“如何回答”。
RLHF / RLAIF：决定“更偏好怎样回答”。
```

## 3. Chat Template：JSON 不是模型真正看到的东西

### 3.1 `messages` 会被渲染为 token 序列

应用层输入常是：

```json
{
  "messages": [
    {"role": "system", "content": "你是一个严格助手。"},
    {"role": "user", "content": "解释 Prefix Cache。"}
  ]
}
```

但模型最终看到的是 Chat Template 渲染后的文本和 token 序列，而不是 JSON 对象本身：

```text
messages
→ apply_chat_template()
→ prompt text
→ tokenizer
→ token IDs
→ model
```

Hugging Face 文档明确说明：chat 的 `role` / `content` 字典会被转换为 token sequence，通常包含 `<|user|>`、`<|assistant|>`、`<|end_of_message|>` 一类控制 token；不同模型可以使用不同格式，即使它们来自同一基础模型。

来源：[Hugging Face Chat Templates](https://huggingface.co/docs/transformers/chat_templating)

同一份抽象对话，放进不同模型时，实际 prompt 可以是完全不同的样子。令 `S` 是 system 内容，`U` 是 user 内容，`A` 是上一轮 assistant 内容：

```text
Qwen / ChatML 风格
<|im_start|>system
S<|im_end|>
<|im_start|>user
U<|im_end|>
<|im_start|>assistant

Mistral Instruct 风格
<s>[INST] U [/INST] A</s> [INST] V [/INST]

Phi-3 风格
<|system|>
S<|end|>
<|user|>
U<|end|>
<|assistant|>
```

业务层三者都可以叫 `role="user"`，但模型侧的边界 token、assistant 的生成起点与历史轮次组织方式不同。错误地混用模板与 tokenizer，有时不会抛出异常，却会让模型在错误的控制序列上续写。

对公开权重，最直接的核验方法不是猜模板，而是用模型自己的 tokenizer 渲染它：

```python
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-7B-Instruct")

prompt = tokenizer.apply_chat_template(
    messages,
    tools=tools,
    tokenize=False,
    add_generation_prompt=True,
)

print(prompt)
```

这里打印出来的是模板渲染后的 prompt text；再经过 tokenizer 才得到实际 token IDs。云端服务仍可能在外侧追加产品策略或默认提示，但这段代码至少让公开模型的模板层不再是黑箱。

### 3.2 Qwen 的公开工具模板

Qwen2.5-Instruct 的公开 `tokenizer_config.json` 可看到：

```text
<|im_start|>
<|im_end|>
<tool_call>
</tool_call>
```

公开模板会在存在 `tools` 时，把工具 schema 放入 system 区段，大意如下：

```text
<|im_start|>system
[system prompt]

# Tools
<tools>
{tool schema 1}
{tool schema 2}
</tools>

[function call format instructions]
<|im_end|>

<|im_start|>user
[user message]
<|im_end|>

<|im_start|>assistant
```

而工具调用输出被包装为：

```text
<tool_call>
{"name":"...","arguments":{...}}
</tool_call>
```

工具执行结果也必须回写进模型协议，而不是只留在应用日志中。公开模板使用相应的工具结果区段，例如：

```text
<tool_response>
{"temperature":28}
</tool_response>
```

下一次模型调用才会同时看到先前的调用与这份结果，并据此继续生成。

来源：[Qwen2.5-Instruct tokenizer configuration](https://huggingface.co/Qwen/Qwen2.5-7B-Instruct/blob/main/tokenizer_config.json)

### 3.3 DeepSeek-V3.1 的公开 thinking 与工具 marker

DeepSeek-V3.1 公开模板包括：

```text
<｜Assistant｜><think>
...
<｜tool▁calls▁begin｜>
...
<｜tool▁calls▁end｜>

<｜tool▁output▁begin｜>
...
<｜tool▁output▁end｜>
```

这意味着 `<think>`、`</think>` 与工具 marker 是模型训练协议的一部分。它们首先是 token / token 序列前缀，并非独立于模型的“硬件推理开关”。

来源：

- [DeepSeek-V3.1 chat template](https://huggingface.co/deepseek-ai/DeepSeek-V3.1/blob/main/assets/chat_template.jinja)
- [DeepSeek-V3.1 model card](https://huggingface.co/deepseek-ai/DeepSeek-V3.1)

### 3.4 Chat Template 能否随意改

#### 技术上：可以改

Jinja 模板可以决定：

- system / user / assistant 的渲染顺序；
- 工具定义如何注入；
- tool result 如何插入；
- 是否加 generation prompt；
- 是否添加自定义 role 或 marker。

Hugging Face 支持通过 `apply_chat_template()` 使用模型提供的模板；推理框架也可能允许指定自定义模板。

#### 模型行为上：不能随意改

模型后训练时见过的是某种 token 分布。若把模板改成模型从未见过的协议，常见结果不是“报错”，而是：

- 角色遵循下降；
- 工具调用格式漂移；
- 结构化输出变差；
- thinking marker 行为失控；
- system prompt 权重变化；
- 工具名称或参数幻觉。

Hugging Face 文档明确提醒：不同 chat 模型使用不同控制 token，错误控制 token 会导致表现显著变差。

来源：[Hugging Face Chat Templates](https://huggingface.co/docs/transformers/chat_templating)

**结论：**Chat Template 是模型协议的一部分，不是 UI 皮肤。

### 3.5 不要把它笼统叫作“token 协议”

“token 协议”不是统一的行业术语，但作为工程简称有用。更准确地说，模型输入输出由五层 contract 叠加而成：

1. **Tokenizer contract**：文本怎样切成 token ID。
2. **Special-token contract**：BOS、EOS、role marker、`<think>` 等 token 的语义。
3. **Chat-template contract**：system、user、assistant、tool result 的拼接顺序与边界。
4. **Tool-serialization contract**：schema、调用和结果使用 JSON、XML 还是专用 marker 表达。
5. **Decoding contract**：stop token、tool choice、grammar、JSON Schema 在生成时怎样限制输出。

它们共同解释了为什么“同一个 `messages` 数组”并不是同一个模型输入。模板错误有时不会报错，却会静悄悄地改变角色边界、工具格式和输出倾向。

DeepSeek-V3.1 的公开模板是一个直观例子：它定义了 thinking 与非 thinking 的不同前缀，也定义了工具调用和工具输出的专用 marker。`<think>` 不是硬件开关；它是模型在后训练轨迹中反复见过的 token 前缀，会把后续生成推向相应格式。将这一套 marker 直接套到其他模型上，通常不是“换一层皮”，而是在改变输入分布。

来源：[DeepSeek-V3.1 chat template](https://huggingface.co/deepseek-ai/DeepSeek-V3.1/blob/main/assets/chat_template.jinja)

### 3.6 Responses API 改的是外形，还是模板？

Chat Completions 与 Responses API 的请求形状确实不同：前者以 `messages` 和 `tools` 为中心，后者以 `instructions`、`input` 和一组 output items 表示对话记录。但对 vLLM 当前的非 Harmony 路径，二者不会因此进入两份模型模板。

#### 两种 API 的对象形状

Chat Completions 将对话和工具直接组织为 message 列表：

```json
{
  "messages": [
    {"role": "system", "content": "You are helpful."},
    {"role": "user", "content": "查北京天气"}
  ],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "get_weather",
        "parameters": {"type": "object"}
      }
    }
  ]
}
```

Responses API 则把 instruction、输入和后续输出 item 分开表达，函数工具 schema 也是扁平形状：

```json
{
  "instructions": "You are helpful.",
  "input": [
    {"role": "user", "content": "查北京天气"}
  ],
  "tools": [
    {
      "type": "function",
      "name": "get_weather",
      "parameters": {"type": "object"}
    }
  ]
}
```

多轮时，Responses 的 output 又会把普通 message、reasoning、`function_call` 与 `function_call_output` 作为独立 item 保存。这种差别是真实的 API 对象差别，不是文档排版差别。

#### vLLM 在中间补了一层会话重建

对非 Harmony 模型，vLLM 的关键路径可概括为：

```text
Responses API 的 input / instructions / output items
        ↓
先归一化成类似 Chat Completions 的 messages + tools
        ↓
走共享的 chat rendering 路径
        ↓
使用该模型原本的 Chat Template
        ↓
token IDs -> 推理
```

按本地核验的 `e6fc848` 源码整理，核心调用形状如下。参数名保持源码含义，省略了与这一判断无关的配置：

```python
tool_dicts = construct_tool_dicts(request.tools, request.tool_choice)

messages = construct_input_messages(
    request_instructions=request.instructions,
    request_input=request.input,
    prev_msg=previous_messages,
    prev_response_output=previous_output,
)

_, engine_inputs = await self.openai_serving_render.preprocess_chat(
    request,
    messages,
    default_template=self.chat_template,
    default_template_content_format=self.chat_template_content_format,
    default_template_kwargs=chat_template_kwargs,
    tool_dicts=tool_dicts,
    tool_parser=tool_parser,
    reasoning_parser=reasoning_parser,
)
```

这段伪代码说明两件事。第一，Responses API JSON 不会被原样塞给模型；第二，归一化完成后，模型仍通过原有模板进入 tokenization。所谓“Responses 兼容”首先是 adapter 的工作，不是另造一门模型语言。

![Chat Completions 直接进入共享 renderer；Responses API 先归一化为 messages 与 tools，再进入同一 renderer、模型原生 Chat Template 和 token IDs。GPT-OSS 使用独立的 Harmony 路径。](/assets/responses-api-normalization.png)

*图 2. API 的对象模型可以不同；模型最终依赖的对话协议不必因此改变。关键步骤是把 Responses 的 item transcript 恢复为模板认识的消息结构。*

在本地核验的 vLLM `e6fc848` 中，Responses 处理器先调用 `construct_tool_dicts()` 和 `construct_input_messages()`，再把结果交给共享的 `preprocess_chat()`。后者将 `messages`、`tools` 和模板参数交给 renderer，最后才产出 prompt token IDs。对应源码见 [`responses/serving.py`](https://github.com/vllm-project/vllm/blob/e6fc848d4ff2d00759516b0c7a47ba640d44e4b2/vllm/entrypoints/openai/responses/serving.py#L598-L619) 与 [`serve/render/serving.py`](https://github.com/vllm-project/vllm/blob/e6fc848d4ff2d00759516b0c7a47ba640d44e4b2/vllm/entrypoints/serve/render/serving.py#L521-L575)。

这层归一化不是字段改名。它需要保住对话状态：

| Responses item | 归一化后的 chat 语义 |
| --- | --- |
| `instructions` | 一条 system message。 |
| 文本 `input` | user message。 |
| `function_call` | 合并到相邻 assistant message 的 `tool_calls`。 |
| `function_call_output` | 一条 `role="tool"` message。 |
| reasoning / assistant item | assistant message 的 reasoning 或 content 部分。 |

把一次工具回合缩成最小形状，会更容易看出为什么不能只改字段名：

```json
// Responses item transcript（示意）
[
  {"type": "reasoning", "content": [{"text": "需要先查天气"}]},
  {
    "type": "function_call",
    "call_id": "call_1",
    "name": "get_weather",
    "arguments": "{\"city\":\"Beijing\"}"
  },
  {
    "type": "function_call_output",
    "call_id": "call_1",
    "output": "{\"temperature\":28}"
  }
]
```

```json
// 归一化后的 chat state（示意）
[
  {
    "role": "assistant",
    "reasoning": "需要先查天气",
    "tool_calls": [
      {
        "id": "call_1",
        "type": "function",
        "function": {
          "name": "get_weather",
          "arguments": "{\"city\":\"Beijing\"}"
        }
      }
    ]
  },
  {
    "role": "tool",
    "tool_call_id": "call_1",
    "content": "{\"temperature\":28}"
  }
]
```

例如，单独出现的 `function_call` 不能简单保留成一条独立消息；许多模型模板假设 tool call 与该 assistant turn 的 content 或 reasoning 共处同一消息对象。vLLM 的 [`responses/utils.py`](https://github.com/vllm-project/vllm/blob/e6fc848d4ff2d00759516b0c7a47ba640d44e4b2/vllm/entrypoints/openai/responses/utils.py#L79-L245) 正在处理这类合并与转换。

因此更准确的结论是：**切换到 Responses API 通常不要求另写 Chat Template；要求的是 adapter 还原出原模板能够正确解释的会话形状。**同一模板仍可能因 `continue_final_message`、`add_generation_prompt`、`reasoning.effort`、`enable_thinking` 或自定义模板参数走到不同的 Jinja 分支，见 [`ResponsesRequest.build_chat_params()`](https://github.com/vllm-project/vllm/blob/e6fc848d4ff2d00759516b0c7a47ba640d44e4b2/vllm/entrypoints/openai/responses/protocol.py#L288-L325)。

真正需要考虑替换模板的情形也与 endpoint 无关：模型原生模板没有工具协议、工具 parser 与模板不匹配，或业务主动引入了模型未训练过的动态工具格式。此时即使继续使用 Chat Completions，也会遇到同一问题。

GPT-OSS 是明确例外。vLLM 对它不走普通 Jinja `preprocess_chat()`，而是走 Harmony message representation、`render_for_completion()` 和 Harmony parser 的专用路径；并且当前 Responses 的 Harmony 分支只接受 `tool_choice` 为 `auto` 或 `none`。这不是“给 GPT-OSS 换一份 Jinja 模板”，而是模型原生输入输出协议本身不同，见 [`_make_request_with_harmony()`](https://github.com/vllm-project/vllm/blob/e6fc848d4ff2d00759516b0c7a47ba640d44e4b2/vllm/entrypoints/openai/responses/serving.py#L728-L746)。

## 4. 工具调用：协议、训练、parser 与约束解码

### 4.1 工具调用并不只是一个 JSON

完整循环通常是：

```text
工具定义 / schema
        ↓
Chat Template 将其渲染入上下文
        ↓
模型选择是否调用工具
        ↓
模型输出 tool call
        ↓
parser 提取调用
        ↓
应用或服务端执行工具
        ↓
tool result 追加回上下文
        ↓
模型继续生成
```

因此工具调用同时涉及：

- 模型是否见过该工具协议；
- 工具描述如何写；
- schema 如何被序列化；
- tool parser 怎样识别调用；
- 工具结果以什么 role / marker 回写；
- 是否并行调用；
- 超时、重试、权限和审计；
- 输出是否使用 constrained decoding。

### 4.2 Claude 工具使用：公开到什么程度

Anthropic 文档公开说明：

- client tools 由你的应用执行；
- server tools 由 Anthropic 基础设施执行；
- 模型产生 `tool_use`，应用回传 `tool_result`，形成 Agent loop；
- strict tool use 可要求工具调用符合 schema。

来源：[Anthropic Tool Use documentation](https://platform.claude.com/docs/en/agents-and-tools/tool-use/overview)

公开资料还确认：启用工具时，服务端会加入工具使用所需的 system prompt，并把格式规则、工具 JSON Schema、用户 system prompt 与工具配置纳入其中。它给出的不是一份可依赖的完整逐字 prompt，而是如下结构性证据：

```text
[formatting instructions]
[tool definitions in JSON Schema]
[user system prompt]
[tool configuration]
```

一个 client tool 的定义与一次模型发起的调用，分别具有可观察的对象形状：

```json
// 请求中声明的工具
{
  "name": "get_weather",
  "description": "查询指定地点天气",
  "input_schema": {
    "type": "object",
    "properties": {
      "location": {"type": "string"}
    },
    "required": ["location"]
  }
}
```

```json
// 模型返回的 tool_use content block
{
  "type": "tool_use",
  "id": "toolu_example",
  "name": "get_weather",
  "input": {"location": "San Francisco, CA"}
}
```

应用执行工具后，必须将结果作为与这次调用关联的 input 回送；模型并不会因为输出了 `tool_use` 就自行访问外部 API：

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_example",
  "content": "{\"temperature\": 18}"
}
```

这一来一回说明 tool calling 的最小闭环不是“模型输出一个 JSON”，而是模型协议、应用执行和结果回写共同构成的状态机。

这足以说明一个边界：模型不是“看见 API 的 tools 字段后自由发挥”；工具定义和工具模式会经过服务端的提示编译或等价机制，再进入模型上下文。不能从这点进一步推断其内部 prompt 的完整措辞、所有模型版本的 parser 或生产调度实现。

**未公开 / 不应假定：**Claude 的完整内部 system prompt、完整工具编排提示词、所有模型版本的 parser 实现和生产推理运行时并没有以可复现方式完整公开。

### 4.3 Structured Outputs / constrained decoding

约束解码的基本思想：

```text
模型 logits
        ↓
JSON Schema / grammar / regex / finite-state constraint
        ↓
过滤非法 token
        ↓
仅从合法 token 路径中采样
```

它常能保证：

- JSON 合法；
- required 字段存在；
- 字段类型匹配；
- enum 值合法；
- 调用受限工具；
- 输出可被 parser 解析。

它不能保证：

- 工具选对；
- 参数语义正确；
- 事实真实；
- 权限合理；
- 用户意图理解正确；
- 推理质量高。

vLLM 文档列出了 Structured Outputs：JSON、grammar、regex、choice、structural tag 等。

来源：[vLLM Structured Outputs](https://docs.vllm.ai/en/latest/features/structured_outputs/)

vLLM 对 named function calling 与 `tool_choice="required"` 使用 structured outputs，并明确说其保证是“validly-parsable function call”，不是“high-quality one”。

来源：[vLLM Tool Calling](https://docs.vllm.ai/en/latest/features/tool_calling/)

一个 JSON Schema 约束请求的最小形状如下：

```python
response = client.chat.completions.create(
    model="Qwen/Qwen2.5-7B-Instruct",
    messages=[
        {"role": "user", "content": "提取姓名和年龄"},
    ],
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "person",
            "schema": {
                "type": "object",
                "properties": {
                    "name": {"type": "string"},
                    "age": {"type": "integer"},
                },
                "required": ["name", "age"],
                "additionalProperties": False,
            },
        },
    },
)
```

这里 schema 约束的是输出空间，不是事实来源。模型能稳定给出 `{"name": "...", "age": 18}` 的合法形状，仍可能从错误文本中抽取、猜错年龄，或根本误解问题。

### 4.4 Assistant prefill 与 constrained decoding 的区别

Assistant prefill，也可称 prefix forcing，是预先放入部分 assistant 内容，让模型从那个前缀继续写。

例如：

```text
assistant:
{"city":"北京","weather":"
```

随后模型从最后一个引号后续写。

vLLM 的 `continue_final_message=true` 将最终消息保持 open-ended，不添加 EOS，模型会继续该消息，而不是启动新的 assistant message。

来源：[vLLM OpenAI-Compatible Server](https://docs.vllm.ai/en/latest/serving/online_serving/openai_compatible_server/)

一个 prefix forcing 请求会明确将 assistant 的最后一条内容保持未闭合：

```python
response = client.chat.completions.create(
    model="Qwen/Qwen2.5-7B-Instruct",
    messages=[
        {"role": "user", "content": "给我一段 JSON，描述北京天气"},
        {"role": "assistant", "content": '{"city":"北京","weather":"'},
    ],
    extra_body={
        "continue_final_message": True,
        "add_generation_prompt": False,
    },
)
```

模型将从 `{"city":"北京","weather":"` 后继续，而不是渲染一个新的 assistant turn。`continue_final_message` 与 `add_generation_prompt` 不能同时为真：前者表示继续已有消息，后者表示开启新消息。

两者的关系：

```text
assistant prefill：固定从哪里开始生成。
constrained decoding：每一步限制哪些 token 能生成。
```

二者可组合：

```text
assistant prefill
+
JSON Schema / grammar
```

## 5. vLLM 推理运行时：Prefill、Decode、KV Cache 与 GQA / MQA

### 5.1 Batching 先改变排队，再改变体验

单请求逐 token 生成很难让 GPU 保持高利用率。batching 会把多个请求放入一次或连续多次 GPU 计算；continuous batching 则在 decode 间隙移出已完成请求、加入新请求，而不是等整批最慢请求结束。

两种调度的差别可以画成：

```text
静态 batch
请求 A  ─────────────────────────────── 完成
请求 B  ─────────── 完成，但仍占着 batch slot
请求 C  ──────────────────────── 完成
                 ↓
             整批释放

continuous batching
decode step 1: [A, B, C]
decode step 2: [A, C, D]   B 完成后，D 立刻进入
decode step 3: [A, D, E]   C 完成后，E 立刻进入
```

它通常首先改变的是系统体验，而不是模型定义：吞吐、排队时间、TTFT、ITL 与流式节奏都会变化。高并发下，同一个模型可能因队列与调度看起来“变慢”或更容易超时；这不等于它的权重发生变化。

### 5.2 Prefill 与 Decode

设输入 prompt 长度为 `P`，模型输出长度为 `Q`。

#### Prefill

Prefill 处理整段 prompt，并建立 KV Cache。

```text
P 个输入 tokens
→ 批量矩阵计算
→ 生成历史 K / V 状态
```

它可同时处理很多 token，通常更接近 compute-bound。

#### Decode

Decode 逐 token 生成：

```text
历史上下文
→ 生成 1 个 token
→ 再生成下一个 token
```

每一步都要读取历史 KV Cache，并读取模型权重；因此 decode 常更接近 memory-bandwidth-bound。

工程直觉：

```text
Prefill：大矩阵乘法，容易利用 GPU 计算能力。
Decode：每步计算较小，但读历史 KV 与权重很多，容易受显存带宽限制。
```

vLLM 的优化文档把 prefill / decode 的不同资源特征作为调度与优化的重要背景。

来源：[vLLM Optimization and Tuning](https://docs.vllm.ai/en/stable/configuration/optimization/)

### 5.3 KV Cache 的容量估算

对 decoder-only Transformer，一个常用近似式：

```text
M_KV ≈ B × T × L × 2 × H_KV × D_head × b
```

其中：

- `B`：同时驻留的序列数；
- `T`：每条序列的当前上下文长度；
- `L`：Transformer 层数；
- `2`：Key 和 Value 两份缓存；
- `H_KV`：KV heads 数；
- `D_head`：每个 head 的维度；
- `b`：每个值的字节数，例如 FP16 / BF16 通常约 2 bytes。

因此：

```text
上下文翻倍 → KV Cache 大致翻倍。
并发翻倍   → KV Cache 大致翻倍。
```

而在 decode 过程中，历史上下文越长，每生成一个 token 需要读取的历史 K/V 越多，所以不仅容量压力增大，显存带宽压力也增大。

将公式代入一个具体量级会更直观。假设模型有 80 层、8 个 KV heads、`head_dim=128`，KV 使用 BF16：

```text
每 token KV
= 2 (K 和 V) × 80 × 8 × 128 × 2 bytes
= 327,680 bytes
≈ 320 KiB

32K tokens  ≈ 10 GiB KV Cache
128K tokens ≈ 40 GiB KV Cache
```

这只是单条序列的近似缓存，还没有计算模型权重、workspace、allocator 开销与其他并发请求。于是“支持 128K”与“在生产负载下能并发服务 128K”是两件不同的事。

一个容易被忽略的区别是：模型标称窗口、服务端允许窗口、当前负载下可实际承载的窗口不是同一个数。

```text
T_usable <= min(T_model, T_server, T_memory_under_load)
```

- `T_model` 受位置编码、长上下文训练与评测约束。超过它有时会报错，有时仍能计算但质量进入分布外区域。
- `T_server` 是部署配置，通常同时约束输入和输出 token。
- `T_memory_under_load` 受模型权重、KV Cache、并发、workspace 与调度策略限制。

因此，“模型支持 128K”并不推出“该服务在任意并发下都能可靠处理 128K”。

将三者拆开看，线上现象才不会显得矛盾：

```text
模型发布为 128K
服务配置 max_model_len = 32K
用户输入 30K，期望输出 8K

结果：30K + 8K > 32K
服务可能拒绝、截断输入、压缩历史或缩短输出；
即使模型权重理论上支持 128K，也不能绕过服务配置与当下显存预算。
```

反过来，将部署长度硬设得高于模型经过训练或位置编码扩展的窗口，也不是“解锁了更多上下文”。有些模型会直接失败；有些模型能计算，却在未训练的位置范围内出现检索、注意力或长程关联质量下降。能运行与值得依赖是两个条件。

### 5.4 MHA、GQA、MQA

#### MHA：Multi-Head Attention

```text
Hq = 32
Hkv = 32
```

每个 Query head 有独立的 K/V。

#### GQA：Grouped-Query Attention

```text
Hq = 32
Hkv = 8
```

多个 Query heads 共用一组 K/V。

#### MQA：Multi-Query Attention

```text
Hq = 32
Hkv = 1
```

所有 Query heads 共用同一组 K/V。

核心关系：

```text
KV Cache ∝ H_KV
```

若 `Hq = 32`，三种设计的缓存关系更直观：

```text
MHA: Hkv = 32
Q1 -> K1,V1   ...   Q32 -> K32,V32

GQA: Hkv = 8
Q1-Q4 -> K1,V1   ...   Q29-Q32 -> K8,V8

MQA: Hkv = 1
Q1-Q32 -> K1,V1
```

在层数、head dimension、精度、上下文和并发都相同时：

```text
MHA KV 体积 = 32
GQA KV 体积 = 8   -> 相对减少约 75%
MQA KV 体积 = 1   -> 相对减少约 96.9%
```

因此：

```text
MHA：KV 最大，表达力最充分。
GQA：质量与速度 / 内存之间的折中。
MQA：KV 最省，但共享最强，质量可能更受影响。
```

原始参考：

- [Multi-Query Attention paper](https://arxiv.org/abs/1911.02150)
- [Grouped-Query Attention paper](https://arxiv.org/abs/2305.13245)

### 5.5 量化与 speculative decoding：性能优化也可能进入行为边界

量化以更低精度表示权重、激活或 KV Cache，换取更低显存占用和可能更高的吞吐。它不是无条件的“免费加速”：数值近似会改变 logits，自回归生成会放大微小差异；因此不同量化配置未必逐 token 一致，结构化输出、代码和工具参数应单独评估。

| 变化 | 常见收益 | 必须单独验证的风险 |
| --- | --- | --- |
| BF16 / FP16 -> FP8 / INT8 / INT4 等量化 | 降低权重或缓存显存，可能提高吞吐。 | logits 改变，低概率分支、代码、JSON 和工具参数可能变化。 |
| KV Cache 量化 | 用更少显存容纳更长上下文或更高并发。 | 长上下文质量与 decode 稳定性需要实测。 |
| speculative decoding | 降低逐 token decode 的等待。 | 收益依赖 draft、接受率和并发；不等于小模型代替大模型。 |

speculative decoding 的基本形状是：

```text
draft model 先猜 k 个 token
        ↓
target model 一次验证这 k 个 token
        ↓
接受的 token 一次前进多个位置
拒绝的位置由 target model 校正
```

严格的接受/拒绝算法可以保持目标模型的采样分布；具体实现的 draft 模型、接受策略与并发负载则决定实际延迟收益。它解决的是 decode 的等待，不是让小模型取代大模型。

### 5.6 流式、超时与停止也是服务契约

streaming 的 chunk 不是模型 token。它可能是半个文本片段、多个 token、工具参数增量、usage 或结束事件。即使两个服务都使用 SSE，也必须分别验证首包时机、tool-call 参数分段、usage 位置、finish reason、异常事件和断连取消。

一个工具调用在流里可能被拆成这样的事件序列：

```text
event 1: assistant role / 空 delta
event 2: tool_call.name = "get_weather"
event 3: tool_call.arguments += "{\"city\":"
event 4: tool_call.arguments += "\"Beijing\"}"
event 5: finish_reason = "tool_calls"
```

因此流式客户端不能把每个 chunk 都当作完整 JSON 或完整 token；必须按调用 ID 聚合增量字段，等到结束事件再交给 parser 或工具执行器。

还要区分三类停止：网络/队列超时、上下文超限处理、生成停止。它们分别可能来自网关或客户端预算、服务端截断或拒绝策略、EOS / `max_tokens` / stop sequence / 安全策略。OpenAI-compatible API 并不会把这三类行为自动标准化。

## 6. Prefix Cache：精确前缀复用，而不是语义缓存

### 6.1 基本机制

KV Cache 在单请求生成期间复用历史状态。

Prefix Cache 则把过去请求已经计算过的 KV blocks 留下来，供未来请求复用。

```text
请求 A：
system + tools + 长文档 + 用户问题 A

请求 B：
system + tools + 长文档 + 用户问题 B
```

若前缀完全一致，B 可复用：

```text
KV(system + tools + 长文档)
```

只重新 prefill 后面的差异部分。

### 6.2 它匹配什么

vLLM 的 Prefix Cache 设计使用 block hash、parent hash 与额外 hash 信息来标识可复用 block。

来源：[vLLM Automatic Prefix Caching design](https://docs.vllm.ai/en/stable/design/prefix_caching/)

这意味着 Prefix Cache 主要是：

```text
完全相同的 token 前缀匹配
```

不是：

```text
语义相似度匹配
```

所以：

```text
“北京今天天气”
```

与：

```text
“今日北京天气如何”
```

即使语义接近，也通常不共享同一 KV 前缀。

这里要把两个概念拆开：**匹配规则是精确的，命中结果是不确定的。**只要 token 前缀和相关 hash 条件一致，复用不是概率判断；但真实请求能否命中，取决于前缀是否稳定、是否落在同一 worker、缓存是否仍在，以及它是否已被更热的工作集挤出。

Prefix Cache 也只省共享前缀的 prefill，不会替请求生成后续回答。若回答本身很长，decode 仍是另一笔成本。

### 6.3 普通 vLLM Prefix Cache 的“过期时间”

**公开事实：**普通 Automatic Prefix Caching 不应简单理解为固定 TTL cache。

其留存更多受：

- 可用 KV Cache 显存；
- 新请求压力；
- LRU 类淘汰；
- block 引用状态；
- worker 生命周期；
- 进程重启；
- 路由位置；

影响。

也就是说：

```text
不是“缓存 10 分钟”。
更像“在显存允许且未被更热请求挤掉时继续保留”。
```

这个结论限定在普通本地 Automatic Prefix Caching。不要把它与分离式 Prefill/Decode 或远端 KV transfer 的租约 TTL 混为一谈：后者管理的是节点间 KV 的传输和等待生命周期，前者管理的是本地显存中可复用 block 的淘汰。

影响普通 APC 存活与命中的变量也不是一个 TTL 参数，而是一组容量和工作负载条件：

| 参数或条件 | 影响 |
| --- | --- |
| `enable_prefix_caching` | 是否启用前缀复用。 |
| KV Cache 显存预算 | 可留存的 block 工作集大小。 |
| `max_model_len` | 单序列潜在 KV 占用。 |
| `max_num_seqs` | 并发越高，淘汰压力越大。 |
| `kv_cache_dtype` | 更低精度可容纳更多 KV，但需要评估质量取舍。 |
| block size | 只复用完整 block，粒度影响可共享范围。 |
| `cache_salt` | 影响哪些请求可以共享同一缓存命名空间。 |

### 6.4 Prefix Cache 的常见失效原因

- system prompt 多一个空格；
- tool schema 字段排序变化；
- JSON 序列化不稳定；
- 在前缀中插入日期、随机 ID、会话变量；
- 模型 / LoRA / tokenizer / template 版本变化；
- 路由到了不同 worker；
- 缓存空间不足被淘汰；
- 前缀只匹配到不完整 block。

**工程建议：**如果重视 Prefix Cache，优先稳定公共 system prompt、工具 schema 序列化与上下文顺序；把高变化字段尽量放在后缀。

## 7. 多租户 Prefix Cache：性能、安全与公平性的权衡

### 7.1 多用户会让 Prefix Cache 失效吗

**不一定。**关键不是“用户数量”，而是“热点前缀工作集”是否能放进缓存。

可以把问题写成容量 `C` 与可复用工作集 `W` 的关系：

```text
C = 当前 worker 可留给可复用 KV blocks 的容量
W = 最近一段时间内、未来还会被复用的前缀总量

W <= C  -> 热点前缀可以留下，多用户可能提高命中率
W >> C  -> 旧 block 反复被挤掉，后续 Agent turn 需要重新 prefill
```

这也解释了为什么 Agent 场景更敏感：工具调用等待期间，会话暂时没有引用它的 KV block；全局 LRU 不知道它稍后会恢复，只知道当前 block 可以被淘汰。若下一轮又路由到另一 worker，即使原 worker 的缓存还在，也无法天然命中。

#### 容易命中的情况

```text
许多用户共享：
- 相同 system prompt；
- 相同 tool schema；
- 相同公开文档；
- 相同模型与模板。
```

此时多用户反而能让公共前缀更热，提高命中率。

#### 容易失效的情况

```text
每个用户都有：
- 独特长 Agent transcript；
- 私有 RAG 文档；
- 不同工具集合；
- 不同模板或动态前缀；
- 不粘性的路由。
```

此时工作集远大于显存 KV Cache，LRU 会不断挤掉旧前缀。

### 7.2 `cache_salt` 隔离了什么

vLLM 的 `cache_salt` 可以让只有共享同一 salt 的请求共享 Prefix Cache。

来源：[vLLM Automatic Prefix Caching design](https://docs.vllm.ai/en/stable/design/prefix_caching/)

可以理解为：

```text
cache_key = hash(parent_hash, token_block, extra_hash, cache_salt)
```

若 token 前缀相同但 salt 不同：

```text
无法命中同一 Prefix Cache。
```

这是一种**逻辑命名空间隔离**，不必然是硬件层物理显存分区。

#### 它解决的风险

全局共享 Prefix Cache 可能带来 timing side-channel：

```text
cache hit  → 少做 prefill → TTFT 可能更低
cache miss → 重做 prefill → TTFT 可能更高
```

攻击者若可猜测敏感前缀、稳定测量时延，并恰好路由到对应 worker，理论上可能将延迟差作为“缓存是否命中”的信号。

`cache_salt` 用于降低不同信任域之间这种共享命中。

#### 它不自动解决的事情

- 不自动给每个租户保留固定 KV 预算；
- 不自动防御宿主机、GPU 或服务端其他漏洞；
- 不自动处理队列拥塞和吞吐公平性；
- 不自动保证每个 API key 都有相同缓存命中率。

### 7.3 API key 与缓存隔离

**不应假定：**“有 API key”并不自动意味着“缓存按 API key 物理分区”。

更稳妥的实现通常是网关完成鉴权后，按组织、项目或会话生成不可预测的 `cache_salt`，再注入后端请求。

一个常见工程思路：

```text
tenant_id = API key 映射出的组织 / 项目身份
cache_salt = HMAC(gateway_key_material, "kv-cache/v1/" + tenant_id)
```

这样：

- 同组织多个 key 可共享缓存；
- key 轮换不必完全冲掉缓存；
- 原始 API key 不进入模型服务；
- 外部调用者不容易伪造隔离域。

这是**工程建议**，不是 vLLM 内置的唯一方案。

### 7.4 公平性：热点租户会挤掉别人吗

在全局 LRU / 热度偏好的缓存中，答案往往是：**可能会。**

```text
高流量租户：
持续访问自己的长前缀
→ 前缀很热
→ 不易被淘汰

低频租户：
工具调用暂停后再回来
→ 前缀可能变冷
→ 更容易被挤掉
→ 恢复时需要重新 prefill
```

这在全局吞吐视角可能是高效的，但在单租户体验视角并不公平。

实际隔离通常不是二选一，而是按信任域与成本选择：

| 方案 | 缓存共享范围 | 主要代价 |
| --- | --- | --- |
| 完全共享，无 salt | 所有请求可复用相同前缀。 | 命中最高，但跨租户 timing side-channel 风险和公平性最弱。 |
| 组织级 salt | 同组织可共享，跨组织不能命中。 | 公共前缀在每个组织重复缓存，仍竞争同一全局 KV pool。 |
| 用户 / 会话级 salt | 单用户或单会话内共享。 | 隔离更强，但工作集迅速膨胀，命中下降。 |
| 独立 worker / GPU pool | 物理或资源池级隔离。 | 缓存、性能与预算边界清楚，但总体利用率最低。 |

更细的生产设计可以让公共 system prompt 与公开工具 schema 走共享域，组织知识和组织工具走组织域，私人历史与上传文档走会话域。`cache_salt` 只控制“能否命中同一键”，不提供租户 KV 保底预算；配额、公平队列和 sticky routing 仍是网关或调度层的职责。

因此高质量多租户 Serving 往往还需要：

- sticky routing；
- per-tenant concurrency quota；
- queue fairness；
- tenant-aware gateway；
- 不同 trust group 的 `cache_salt`；
- 高价值客户独立 worker pool 或独立 GPU pool。

这些是**工程推断 / 设计建议**，不应误写成“vLLM 默认自动提供”。

## 9. OpenAI Tool Search：为什么它更利于 Prefix Cache

### 9.1 公开事实

OpenAI Tool Search 文档说明：模型可以按需搜索并加载 deferred tools，避免在一开始把全部工具定义放入上下文。文档还明确说，为了成本和延迟，Tool Search 设计为尽量保留模型 cache；发现的新工具会被注入 context window 的末尾。

来源：[OpenAI Tool Search](https://developers.openai.com/api/docs/guides/tools-tool-search)

文档当前列明：`tool_search` 仅支持 `gpt-5.4` 及之后模型。

来源：[OpenAI Tool Search](https://developers.openai.com/api/docs/guides/tools-tool-search)

### 9.2 传统工具列表为什么伤害缓存

传统方式：

```text
system
+ 500 个完整 tool schemas
+ user
```

只要工具列表的内容、排序、描述、版本或权限变化，前缀就变化。Prefix Cache 因此可能大面积 miss。

### 9.3 Tool Search 的 append-only 思路

更 cache-friendly 的结构是：

```text
稳定前缀：
system
+ tool-search rules
+ namespace / server 摘要
+ user

动态后缀：
tool_search_call
+ tool_search_output
+ loaded tool schemas
+ actual function_call
```

这样动态加载的 schema 被追加到上下文末尾，而不是回头重写前面的完整工具列表。

**工程推断：**这会把缓存扰动从“大段前缀”缩小到“动态后缀”，因此更有利于保留稳定前缀的 Prefix Cache。

### 9.4 vLLM 是否已完整原生支持 OpenAI Tool Search

**截至本文快照，保守表述如下：**

- vLLM 文档明确支持 Responses API 路由、tool calling、Structured Outputs 和 MCP 相关示例；
- 本文未依据公开 vLLM 文档确认“完整原生复刻 OpenAI hosted / client-executed Tool Search 协议”；
- 因此，若在 vLLM 上实现同类能力，通常仍应准备应用层 Agent runtime 来负责工具搜索、schema 注入与流程编排。

这是一条**保守判断**，未来应随着 vLLM 文档和 release 更新重新验证。

## 10. 动态工具：为什么不能只改 Chat Template

### 10.1 Chat Template 与 Agent runtime 是两件事

```text
Chat Template：
把 messages / tools 渲染为模型输入 token 序列。

Agent runtime：
模型发起搜索
→ 服务端检索工具
→ 返回匹配 schema
→ 追加上下文
→ 再次调用模型
→ 执行真实工具。
```

只写一个新的 Jinja 模板，并不会自动获得：

- 工具检索；
- 工具注册；
- 权限控制；
- schema 版本管理；
- 重试；
- 错误恢复；
- 多轮状态管理；
- tool result 回填。

### 10.2 把动态工具直接塞在后面，模型会不会认识

对于某个具体模型，例如 DeepSeek-V3.1：

- 它可以处理追加在上下文后面的合法文本 token；
- 它通常不会因为“后面出现了工具说明”而立即程序性报错；
- 但公开模板训练的默认协议，未必包含“tool result 里的新 schema 自动成为下一步可调用函数”的语义。

因此结果可能是：

- Demo 中模型看起来能理解；
- 生产中工具选择、参数填写、格式稳定性下降；
- 模型把 schema 当普通文档，而不是新的 callable capability；
- 模型仍倾向调用最初工具列表中出现的工具。

这叫做**协议分布偏移**。

DeepSeek-V3.1 的公开模板还揭示了一个容易忽略的缓存陷阱：它会把 system messages 聚合到 prompt 前部。若把新发现的工具 schema 作为一条新的 system message 加回去，看似是在“后面追加”，最终渲染却可能变成：

```text
第一轮渲染：
system + 工具 A + user

第二轮业务代码新增一条 system message：工具 B
        ↓
模板重新聚合 system messages
        ↓
system + 工具 A + 工具 B + 历史消息
```

此时稳定前缀已经被重写，Prefix Cache 自然无法保住。真正的 append-only 必须借助模型原生的后续消息通道，例如 tool result：

```text
稳定前缀
+ user
+ assistant: tool_search(...)
+ tool output: 已加载工具 schema
+ assistant: 下一次生成
```

从 token 顺序看，这才是追加；但模型是否把 tool output 中的 schema 理解成“新注册的 callable capability”，仍然是训练与评测问题，而不是模板是否渲染成功的问题。

### 10.3 是否需要专门训练动态工具轨迹

**结论：**

- 做 Demo：可能依靠 in-context learning 就能工作；
- 做生产 Agent：最好有训练数据、足够多示例或专项评测。

更理想的训练轨迹：

```text
用户请求
→ assistant: tool_search("CRM refund")
→ tool output: loaded_tools = [create_refund, get_refund_status]
→ assistant: create_refund(...)
→ tool output: refund created
→ assistant: final response
```

模型需要学到的不是“后面出现一段 JSON”，而是：

```text
tool_search output
= 新的可调用能力集合
```

### 10.4 更稳的开源实现：固定 meta-tools

对未专门训练动态工具协议的开源模型，更稳的模式是初始只提供固定 meta-tools：

```text
search_tools(query)
invoke_loaded_tool(tool_id, arguments)
```

初始工具面保持固定，动态工具名只作为数据进入第二个 meta-tool：

```json
// 模型在初始模板中已经见过的固定调用
{"name": "search_tools", "arguments": {"query": "customer orders"}}

// tool output 追加到 transcript
{
  "loaded_tools": [
    {
      "id": "crm.list_open_orders",
      "description": "List open orders for a customer.",
      "parameters": {
        "type": "object",
        "properties": {"customer_id": {"type": "string"}},
        "required": ["customer_id"]
      }
    }
  ]
}

// 模型仍调用固定 meta-tool；动态工具名是参数数据
{
  "name": "invoke_loaded_tool",
  "arguments": {
    "tool_id": "crm.list_open_orders",
    "arguments": {"customer_id": "..."}
  }
}
```

流程：

```text
system + 固定 meta-tools + user
        ↓
assistant: search_tools({"query":"customer orders"})
        ↓
tool result: 返回已加载工具及 schema
        ↓
assistant: invoke_loaded_tool({
  "tool_id":"crm.list_open_orders",
  "arguments": {"customer_id":"CUST-123"}
})
        ↓
运行时验证、执行真实工具
```

优点：

- 初始工具定义稳定；
- 模型只调用它在训练中更可能理解的固定工具；
- 动态工具名成为数据，而不是新的 model-level function marker；
- Prefix Cache 更友好；
- 运行时可严格做权限和 schema 校验。

动态 `arguments` 可进一步用 JSON Schema 或 grammar 做约束解码。

## 11. 推荐的生产设计原则

### 11.1 把“业务 API”与“模型协议”分开

```text
业务 API：
messages / tools / response_format / stream

模型协议：
chat template / tokenizer / special tokens / parser / grammar / stop rules
```

API 可以统一，模型协议通常不能直接统一。

### 11.2 记录版本，不只记录模型名

一次线上行为复现至少应记录：

```text
model revision
tokenizer revision
chat template revision
tool parser revision
structured-output backend revision
serving configuration
```

仅记录“Qwen”或“DeepSeek”通常不足以解释行为差异。

### 11.3 为 Prefix Cache 设计稳定前缀

建议：

- 固定 system prompt；
- 统一工具 schema 字段顺序；
- 固定 JSON 序列化；
- 把动态日期、会话 ID、临时权限 token 放后缀；
- 让公共 context 与私有 context 分层；
- 用 sticky routing 保住 worker 局部缓存；
- 监控 TTFT、ITL、cache hit rate、eviction、queue time。

### 11.4 对动态工具做“语义与结构分治”

```text
模型：
理解意图、选择工具、提出参数语义。

约束解码：
保证调用 JSON / schema 合法。

运行时：
权限、审计、执行、超时、重试、异常处理。
```

不要把结构合法误认为 Agent 正确。

### 11.5 多租户缓存要当作安全与产品设计

不要只问：

```text
“Prefix Cache 能快多少？”
```

还要问：

```text
“谁能共享？”
“谁会被挤掉？”
“是否泄露缓存命中信号？”
“是否需要 per-tenant quota？”
“高价值客户是否需要独立 worker pool？”
```

## 12. 最终结论

1. **OpenAI-like API 解决的是接入兼容，不是行为同构。**
2. **模型行为由训练、tokenizer、Chat Template、工具协议、推理运行时与产品策略共同决定。**
3. **Chat Template 是模型协议，不是 UI 装饰。改模板可能不报错，但会静悄悄地让模型表现变差。**
4. **工具调用既是协议问题，也是推理问题。**
5. **Structured Outputs 能管结构，不能保证语义正确。**
6. **KV Cache 决定长上下文的显存账本；GQA / MQA 能直接改变这一账本。**
7. **Prefix Cache 是精确 token 前缀复用，不是语义缓存。**
8. **多租户 Prefix Cache 同时是性能、安全与公平性问题。**
9. **Tool Search 的核心价值之一，是把动态工具加载变成 append-only 后缀，从而尽量保住稳定前缀与模型 cache。**
10. **工程判断应明确区分：公开事实、工程推断与未公开信息。**

## 13. 继续验证的问题

1. vLLM 是否新增对 OpenAI Tool Search hosted / client-executed 协议的明确原生支持。
2. Qwen、DeepSeek、Llama、GPT-OSS 在动态工具协议下的稳定性差异。
3. 动态 tool schema 是否能被 JSON Schema / grammar 覆盖主要参数组合。
4. Prefix Cache 在长 Agent 会话与多租户场景下的真实 hit rate、TTFT、eviction 与 preemption。
5. `cache_salt`、sticky routing、tenant quota、独立 worker pool 的成本与隔离取舍。
6. 模型、tokenizer、chat template、tool parser、structured-output backend 升级后是否改变行为。
7. Tool Search 是真正降低总成本，还是只把成本迁移到后续 Agent 回合。
8. 长上下文下 KV Cache 容量、带宽、量化精度、GQA/MQA 与服务质量的关系。
