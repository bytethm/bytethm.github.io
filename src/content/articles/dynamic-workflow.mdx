一个普通 Agent 最容易理解：用户给它一个目标，它读上下文，想下一步，调用工具，拿到结果，再继续想。

这就是很多 Agent 系统的基本形态。它足够简单，也足够强大。问题通常不是它不会做事，而是任务一变长，所有事情都会被塞回同一个地方：模型上下文。

计划在上下文里，中间结果在上下文里，下一步判断在上下文里，完成判断也在上下文里。模型每一轮都要重新读这堆文本，恢复“现在做到哪了”。

这篇文章讨论的 Dynamic Workflow，就是为了解决这个压力。你可以先把它理解成一句话：

> 让模型先写出“这次任务该怎么推进”，再让运行时按这段流程执行。

它不是为了让 Agent 多一个炫技的脚本能力。它关心的是一个更朴素的问题：复杂任务里，那段“先做什么、后做什么、失败怎么办、什么时候结束”的推进逻辑，到底放在哪里。

整篇文章只围绕这条线展开：

```text
普通 ReAct：模型每一轮临时决定下一步
多 Agent：执行角色变多了，但仍要回答谁来编排
固定 Workflow：人提前把流程写死
Dynamic Workflow：模型为当前任务生成流程，运行时执行流程
```

如果你第一次接触这个概念，先不用记 API 名字。先看懂这条线就够了。

## 先看普通 ReAct 一次怎么跑

普通 ReAct 循环的优点是简单：

```text
模型观察上下文
→ 决定下一步
→ 调用工具或输出答案
→ 结果回到上下文
→ 继续
```

把它放进一个具体任务里，会更容易看清楚。

假设用户说：

```text
帮我比较三个远程协作方案：
A. 每天站会
B. 异步文字更新 + 每周一次决策会
C. 完全按需开会

请先分析，再让 reviewer 检查风险。
如果 reviewer 不通过，就根据反馈重写建议。
```

普通 ReAct 大致会这样推进：

```text
第 1 轮
模型读用户目标
→ 决定先分析方案 A/B/C
→ 输出或调用工具得到一批分析结果
→ 分析结果被追加到上下文

第 2 轮
模型重新读：用户目标 + 第 1 轮分析结果
→ 决定做 review
→ reviewer 结论被追加到上下文

第 3 轮
模型重新读：用户目标 + 分析结果 + review 结论
→ 判断 review 是否通过
→ 如果不通过，生成修订版
→ 修订版继续追加到上下文

第 4 轮
模型重新读所有内容
→ 判断是否可以最终回答
```

这个流程不是错的。它甚至很自然：模型像一个人一样，一边看聊天记录，一边决定下一步。

它的优势也很明显：

- 不需要提前写流程；
- 中途可以根据结果改方向；
- 简单任务可以很快完成；
- 所有内容都在一条对话里，容易解释。

但它也有一个隐含代价：上下文同时承担了太多职责。

## ReAct 的压力：上下文太忙

在上面的例子里，上下文至少要保存五类东西：

| 上下文里的内容 | 它实际承担的职责 |
| --- | --- |
| 用户原始目标 | 任务定义 |
| A/B/C 的分析结果 | 中间数据 |
| reviewer 的意见 | 验收证据 |
| “不通过就重写” | 控制逻辑 |
| 是否已经完成 | 结束判断 |

这些东西全都混在一串 messages 里。模型每一轮都要靠阅读上下文来恢复状态。

短任务问题不大。长任务会出现几个典型压力：

- 中间结果越来越多，模型为了做下一步必须反复读旧内容；
- “先分析、再 review、不通过就重写”这种控制逻辑只是写在 prompt 里，没有真正变成可执行结构；
- review 结果是否被使用，靠模型自己记住；
- 并发分析 A/B/C 只是自然语言要求，不一定真的并发；
- 模型没有工具调用时，系统很容易把它理解成任务结束。

所以 ReAct 的问题不是“不智能”。它的问题是：那段真正推进任务的逻辑没有显式位置。

后面我会把这段逻辑叫作**任务程序**。它不是指某种具体编程语言，而是指：

```text
保存任务状态
选择下一步
调度执行单元
消费中间结果
判断能否结束
```

普通 ReAct 当然也有任务程序。只是这段程序分散在模型每一轮推理里，藏在上下文中。

## 多 Agent 之后，谁来编排

看到 ReAct 压力后，一个自然反应是：那就多找几个 Agent。

比如：

```text
analyst 负责分析
writer 负责编写建议
reviewer 负责检查
```

这一步很合理。长任务往往确实需要不同角色：有人负责分析，有人负责生成，有人负责审查。

但多 Agent 只是把“谁来做事”拆开了，还没有回答“这些 Agent 怎么被串起来”。

真正的问题会马上出现：

```text
analyst 的结果给谁？
writer 什么时候开始？
reviewer 不通过怎么办？
review 意见怎样回到 writer？
重写几次算够？
谁决定最终可以结束？
```

这些不是角色问题，而是编排问题。

可以用一段伪代码看得更清楚：

```text
analysis = analyst(input)
draft = writer(input, analysis)
review = reviewer(draft)

if review.pass:
  return draft

draft = writer(input, analysis, review.feedback)
return draft
```

这里真正让任务跑起来的，不是 analyst、writer、reviewer 三个名字，而是它们之间的边：

```text
analysis → draft → review → feedback → revised draft
```

没有这段控制逻辑，多 Agent 只是一组可调用能力。系统表面上有多个角色，任务程序仍然可能藏在一个 manager prompt 里。

所以要区分两件事：

```text
Agent 解决“谁来做事”
Workflow 解决“事情怎样推进”
```

多个 Agent 出现后，编排它们大致有三种方式：

![三种多 Agent 编排方式：固定 Workflow、root Agent 每轮工具调用、Dynamic Workflow。](/assets/dynamic-workflow-orchestration-modes.png)

*图 1. 多 Agent 不是一种编排方式。真正的差别是控制流归谁所有：业务代码、root Agent 的上下文推理，还是模型生成后交给 runtime 执行的 workflow code。*

| 子 Agent 怎么被串起来 | 谁决定连接关系 | 连接关系在哪里执行 | 对应形态 |
| --- | --- | --- | --- |
| 代码或 Graph 预先串好 | 人 / 业务代码 | 固定 workflow runtime | 固定 Workflow |
| root Agent 每轮工具调用 | LLM 每轮基于上下文决定 | root Agent 的 ReAct loop | LLM-driven orchestration |
| 模型先生成 workflow code | LLM 为当前任务生成任务程序 | workflow runtime 执行代码 | Dynamic Workflow |

第二种最接近普通 ReAct 的延伸。root Agent 可以把 analyst、writer、reviewer 都当成工具调用：

```text
root Agent 读上下文
→ 决定调用 analyst
→ analyst 结果回到 root 上下文
→ root Agent 再决定调用 writer
→ writer 结果回到 root 上下文
→ root Agent 再决定调用 reviewer
→ reviewer 结果回到 root 上下文
→ root Agent 判断是否要重写
```

这种方式很灵活，也很自然。但它的任务程序仍然主要存在于 root Agent 的上下文推理里。子 Agent 变多了，编排本身还没有显式化。

## 预先编排：固定 Workflow

既然任务程序藏在上下文里有问题，那可以把它拿出来，写成固定 workflow。

这里要先澄清一个容易混的词：很多人说 workflow，默认想到的是一串比较固定的 LLM 节点，例如：

```text
extract → classify → summarize → answer
```

这种 workflow 常被拿来和 ReAct Agent 对位：一个是预先写好的 LLM 流程，一个是模型每轮自己决定下一步。

本文说的 workflow 更宽一点。它不只可以编排简单 LLM 节点，也可以把一个完整的 ReAct Agent 当成节点放进去：

```text
Graph / Workflow
├─ LLM node
├─ tool node
└─ Agent node
   └─ 内部仍然可以自己 ReAct、调用工具、维护上下文
```

也就是说，有两层 workflow：

| workflow 用法 | 节点通常是什么 | 主要对位 |
| --- | --- | --- |
| LLM 节点流程 | prompt / LLM call / tool call | 替代一个普通 ReAct Agent 的部分推理过程 |
| Agent 编排流程 | ReAct Agent / AgentTool / Graph node | 编排多个能独立行动的 Agent |

这两种并不冲突。一个框架里的 Graph 既可以只放简单 LLM node，也可以放完整 Agent node。本文后面讨论 Dynamic Workflow 时，重点是第二种：把多个 Agent 或临时角色串成一个任务级流程。

固定 workflow 会提前定义：

- 先走哪个阶段；
- 哪些阶段可以并发；
- 失败后回到哪里；
- 中间结果放在哪里；
- 什么条件下结束。

这对稳定流程非常合适。比如标准审批、固定测试、确定的数据处理管道，都不应该每次让模型重新发明。

但固定 workflow 也有代价：它要求人提前知道流程。

真实 Agent 任务经常不是这样。用户目标可能很粗；是否需要调查取决于现场证据；某一步失败后，下一步可能不是重试，而是换路径；有些任务很简单，根本不值得走完整流程。

当异常路径越来越多，固定图会膨胀。每增加一种情况，就多一个分支；每允许一种绕行，就多一条边。最后得到的不是更灵活的 workflow，而是一张把适用边界藏起来的大图。

所以我们有两个端点：

```text
ReAct：很灵活，但任务程序隐含在上下文里
固定 workflow：很清楚，但要求流程预先确定
```

Dynamic Workflow 站在这两个端点之间：它不要求人提前写死流程，也不让 root Agent 每一轮都从上下文里重新推断流程。

## 动态生成编排：Dynamic Workflow

前面那张表里，最后一种方式就是 Dynamic Workflow。

它可以用一句话描述：

> 模型为当前任务生成一段任务程序，运行时负责执行这段程序。

它和固定 workflow 的区别是：程序不是人预先写死的，而是根据当前任务生成。

它和 ReAct 的区别是：程序一旦生成，循环、分支、并发和阶段结果就不再完全藏在模型上下文里，而是进入一个可执行运行时。

公开产品里已经能看到类似形态。Claude Code 的 Dynamic Workflows 把 workflow 定义为一段由模型生成、runtime 执行的 JavaScript script；官方文档特别强调，loop、branch 和 intermediate results 会留在 script variables 里，而不是全部回到模型上下文。[官方文档](https://code.claude.com/docs/en/workflows)

这些产品细节不是本文的重点。真正值得借鉴的是它验证了一条机制方向：动态工作流不是“多开几个 Agent”，而是把计划从上下文移到可执行程序。

![Dynamic Workflow 把计划、分支、并发和中间状态从 prompt context 移到 workflow runtime；子 Agent 仍在宿主 Agent Runtime 中执行，事件继续进入统一 event stream。](/assets/dynamic-workflow-hero.png)

*图 2. Dynamic Workflow 的核心不是多开几个 Agent，而是把任务程序从模型上下文移到可执行运行时；workflow code 负责协调，Agent Runtime 仍负责子 Agent、session 和事件流。*

可以把三种形态放在一起：

| 形态 | 任务程序从哪里来 | 谁推进下一步 | 优点 | 代价 |
| --- | --- | --- | --- | --- |
| 固定 workflow | 人预先编写 | 代码 / Graph runtime | 稳定、可控、可复用 | 对未知路径不灵活 |
| ReAct | 模型每轮即时解释 | 模型上下文 | 灵活、轻量 | 状态和控制隐式 |
| Dynamic Workflow | 模型为当前任务生成 | workflow runtime | 灵活且可执行 | 需要清楚的运行边界 |

## 一次 Workflow Run 长什么样

继续用前面的远程协作方案例子。一次动态工作流大致是这样：

```text
用户提出复杂任务
→ root Agent 判断普通回答不够，调用 run_workflow
→ 模型生成一段 workflow code
→ workflow runtime 启动一次 workflow run
  ├─ phase 1：并发创建 child agents，分别分析 A/B/C
  ├─ phase 2：创建 reviewer agent，交叉检查三份分析
  ├─ phase 3：如果 reviewer 不通过，带着 feedback 重写
  └─ phase 4：聚合成一个紧凑结果
→ root Agent 读取 workflow result，组织最终回答
```

这不是另开一个和 Agent 系统无关的脚本。workflow run 更像一次临时生成的任务程序实例：它有自己的局部变量和阶段结果，但每次真实的 Agent 调用、工具调用和事件记录，仍然回到宿主 Agent Runtime。

同一件事可以从三个视角看：

| 视角 | 看到的对象 | 主要关心 |
| --- | --- | --- |
| 用户 | 一个复杂任务正在被推进 | 过程是否可信，结果是否有证据 |
| root Agent | 一次 `run_workflow` 工具调用和最终结果 | 是否需要生成任务程序，如何解释结果 |
| workflow runtime | code、局部变量、child agents、阶段结果 | 下一步怎么走，中间结果如何传递 |

这张表很重要。它说明 Dynamic Workflow 不是把 root Agent 替换掉，而是给 root Agent 一个更适合复杂任务的执行单元。

## 这段程序长什么样

前面的描述还是抽象。下面是一段简化后的 workflow code。读这段代码时，不要先纠结 Python 语法，只看它表达的流程：并发分析、集中 review、不通过就重写。

```python
analyses = await parallel([
    lambda: agent({"plan": "daily standup"}, instruction="Analyze this plan."),
    lambda: agent({"plan": "async updates"}, instruction="Analyze this plan."),
    lambda: agent({"plan": "meet on demand"}, instruction="Analyze this plan.")
])

review = await agent(
    {"analyses": analyses},
    instruction="Review the analyses. Return pass/fail and feedback.",
    schema={
        "type": "object",
        "additionalProperties": False,
        "properties": {
            "pass": {"type": "boolean"},
            "feedback": {"type": "string"}
        },
        "required": ["pass", "feedback"]
    }
)

if not review["pass"]:
    revised = await agent(
        {"analyses": analyses, "feedback": review["feedback"]},
        instruction="Revise the recommendation using the feedback."
    )
else:
    revised = await agent(
        {"analyses": analyses},
        instruction="Write the final recommendation."
    )

return {"recommendation": revised["text"], "review": review}
```

这段代码没有什么神秘的地方。它只是把原本写在 prompt 里的要求变成了可执行结构：

```text
并发分析 A/B/C
→ review
→ 不通过就重写
→ 返回结构化结果
```

差别在于，prompt 里的控制结构要靠模型每轮记住；代码里的控制结构由运行时执行。模型仍然负责为当前任务生成流程，但确定下来的顺序、分支和中间变量不必每轮重新塞回自然语言上下文。

## 读到这里，核心已经成立

到这里，Dynamic Workflow 的基本心智模型已经够用了：

```text
普通 ReAct 把推进逻辑留在上下文里
固定 Workflow 把推进逻辑写在预定义流程里
Dynamic Workflow 让模型为当前任务生成推进逻辑，再由运行时执行
```

如果只想建立概念，读到这里已经够了。后面的内容是工程化展开：如果真要把这种机制做进 Agent 框架，怎样让它既有表达力，又不变成失控的脚本执行器。

## 代码不是普通脚本

看到这里，很容易问：既然模型能写代码，那是不是直接让它写一段脚本调用业务接口就够了？

这不是 Dynamic Workflow 最值得保留的形态。

普通脚本直接访问外部能力，会绕过 Agent Runtime。模型配置、工具声明、权限策略、结构化输出、事件流、会话记录、回调和观测链路都会被切断。脚本执行完以后，外层系统只看到一个工具结果；中间到底哪个子 Agent 做了什么、调用了什么工具、失败在哪里，都很难继续作为 Agent 事件被消费。

动态工作流里的代码更像一段受控胶水：

```text
workflow code
→ runtime bridge
→ host Agent / host tool
→ normal Agent Runtime
→ event stream / session / callbacks
→ compact workflow result
```

代码负责表达控制逻辑，但真实能力仍然留在宿主运行时里。子 Agent 仍按框架规则运行；工具仍走原来的 schema、权限和观测；事件仍可以回到统一的 Runner event stream。

这条边界很关键。Dynamic Workflow 不是给模型一台裸脚本机，而是给模型一个可以写任务程序的受控入口。

## `agent(...)` 不是普通函数

动态工作流里的 `agent(...)` 看起来像函数调用，但它不应该被理解成普通函数。

普通函数的身份和行为在代码里已经固定。动态子 Agent 则更像“为当前步骤创建一个临时角色”：

```python
risk_review = await agent(
    {"recommendation": recommendation},
    instruction="Act as a skeptical reviewer. Find missing risks."
)
```

同一个基础模板，可以在不同调用里承担不同角色。一次是分析员，一次是审阅者，一次是最终汇总者。它们可以共享宿主侧已经配置好的模型、工具、权限和回调，但每次调用的 instruction、输入、结构化输出要求可以不同。

这比把一组固定 AgentTool 暴露成函数更灵活。AgentTool 适合稳定角色；动态子 Agent 适合临时角色。前者像“系统里已有一个 reviewer”；后者像“这一步需要一个 reviewer，现在按这个标准临时创建”。

这也是动态工作流和普通[工具编排](../tool-code-orchestration/)的区别之一：工具编排更关注工具之间的数据搬运和类型适配；动态工作流更关注任务里的临时角色、阶段、并发和回路。

## 并发、串行和循环只是基本语法

Dynamic Workflow 里常见的 `parallel`、`pipeline`、loop，不应该先被理解成 API，而应该被理解成任务程序的基本语法。

并发就是“这些事情互不依赖，可以同时做”：

```python
analyses = await parallel([
    lambda: agent({"plan": "A"}, instruction="Analyze this plan."),
    lambda: agent({"plan": "B"}, instruction="Analyze this plan."),
    lambda: agent({"plan": "C"}, instruction="Analyze this plan.")
])
```

串行就是“后一阶段要吃前一阶段的结果”：

```python
result = await pipeline(
    {"topic": topic},
    [
        lambda x: agent(x, instruction="Draft a proposal."),
        lambda x: agent(x, instruction="Review and improve the proposal.")
    ]
)
```

循环就是“检查不通过就带着反馈再做一次”：

```python
for _ in range(2):
    review = await agent(
        {"draft": draft["text"]},
        instruction="Return whether the draft is acceptable and why.",
        schema=review_schema
    )
    if review["pass"]:
        break
    draft = await agent(
        {"draft": draft["text"], "feedback": review["feedback"]},
        instruction="Revise the draft."
    )
```

这些结构原本也可以写成一句 prompt：

```text
请先做 A、B、C，再比较，如果不通过就重写。
```

但 prompt 只是要求；workflow code 是可执行控制结构。

## 质量模式比规模更重要

Dynamic Workflow 很容易被理解成“并发开很多 Agent”。这只是表象。

更有价值的是，它让一些质量模式变成程序结构，而不是 prompt 里的愿望：

```text
多个 Agent 独立寻找问题
→ 另一些 Agent 交叉验证 finding
→ 只保留通过验证的结论
→ 汇总成最终报告
```

或者：

```text
多个 Agent 从不同角度提出方案
→ adversarial reviewer 尝试推翻每个方案
→ 根据证据选择更稳的路线
```

Claude Code 的发布文章里反复强调这一点：workflow 不只是把任务 fan out，而是让结果在进入最终答案前先被独立检查、互相反驳或收敛。[Introducing dynamic workflows](https://claude.com/blog/introducing-dynamic-workflows-in-claude-code)

这对框架设计有启发。一个动态工作流接口如果只暴露“并发调用 Agent”，价值是不完整的。它还应该让模型自然表达 review、vote、refute、retry、filter 这些质量控制结构。规模带来吞吐，质量模式才带来可信度。

## 事件流不是细节

如果动态工作流只是一个黑盒工具，前端和上层系统只能看到：

```text
run_workflow started
run_workflow finished
```

那它会丢掉 Agent Runtime 很重要的价值。

复杂任务往往需要展示过程：哪个子 Agent 正在分析，哪个子 Agent 调用了工具，review 阶段为什么失败，重写发生了几轮。用户和系统都不只关心最终答案，也关心中间证据。

因此，一个好的动态工作流实现应该让子 Agent 的事件继续进入统一事件流：

```text
root Agent
→ tool call: run_workflow
  → child Agent: analyst output
  → child Agent: reviewer tool call
  → child Agent: reviewer result
  → workflow result
→ root Agent final answer
```

这样，workflow 代码虽然在一个临时运行时里执行，但 Agent 本身没有脱离宿主框架。会话记录、事件消费、进度展示和调试工具仍然能看到足够细的过程。

这也是“受控任务程序”和“普通脚本”的差别。普通脚本通常只把最终 stdout 或返回值交回来；动态工作流应该把内部 Agent 执行仍然保留在框架的事件模型里。

公开产品的做法也说明，观测不应只停留在日志。Claude Code 的 workflow view 会按 phase 和 agent 展示进度、耗时、token、最近工具调用和结果，并支持暂停、停止、重启某个 agent。这些细节背后的判断是：长任务如果不可观察，就很难被信任；动态工作流越能跑很久，越不能只给一个最终答案。

因此，动态工作流的事件模型最好从一开始就按“可展开的任务树”设计：

```text
workflow run
├─ phase
│  ├─ child agent
│  │  ├─ model output
│  │  ├─ tool call
│  │  └─ result
│  └─ child agent
└─ workflow result
```

这不是 UI 细节，而是运行时边界。没有这棵树，用户只能相信最终汇总；有了这棵树，用户才能检查过程证据。

## 审批和权限是运行边界的一部分

动态工作流在启动前最好能让用户看到“将要运行什么”。

Claude Code 的文档里，workflow 启动时可以查看计划阶段、查看 raw script、确认或取消；不同 permission mode 下确认策略不同。更关键的是，workflow script 本身不直接读写文件或执行 shell，读写和命令由它调度的 agents 完成；这些 agents 仍继承工具 allowlist，未授权工具仍可能触发权限处理。[官方文档](https://code.claude.com/docs/en/workflows)

这背后的原则比具体交互更重要：

```text
脚本负责协调
Agent 负责行动
工具权限仍归宿主运行时控制
```

如果 workflow code 能直接绕过宿主工具系统访问文件、shell 或外部接口，那它就退化成普通脚本执行器。Dynamic Workflow 的工程价值，恰恰在于它把“模型生成控制逻辑”和“系统授予真实能力”分开。

这也意味着，动态工作流不适合在运行中无限等待用户输入。长任务可以因为权限提示暂停，但如果某个阶段需要业务 sign-off，更稳的做法是把它拆成两个 workflow：前一个产生可审阅结果，后一个在用户确认后继续。

## 一次性的运行，和可维护的资产

动态工作流还有一个容易争论的问题：生成出来的 workflow 要不要保存成文件，后续继续维护和复用？

这个问题对新手可能有点远，但它决定了这个能力最终会变成“一次性工具调用”，还是能沉淀成长期资产。

这要分场景。

一次性 workflow 的优势是轻。每个任务生成一段最贴合当前上下文的程序，运行完就结束。它适合临时角色、临时判断和路径不稳定的任务。

可维护 workflow 的优势是可以积累经验。一个流程被多次验证后，保存下来、人工审阅、逐步修改，可能变成团队资产。

但保存也有代价。一个脚本一旦变成资产，就要回答更多问题：

- 它依赖哪些 Agent 模板和工具？
- 输入参数怎样抽象才不过度复杂？
- 版本怎样管理？
- 运行 workspace 从哪里来？
- 多节点部署时文件怎样保持一致？
- 它和固定 workflow 的边界在哪里？

很多临时 workflow 并不值得沉淀成文件。它们的价值来自“刚好适合这次任务”。强行复用，可能会把模型从“写一段清楚代码”的问题，变成“理解一堆历史参数和隐含约定”的问题。

更稳的路径是分层演进：

```text
先支持一次性 workflow
→ 让运行边界、事件流和能力约束成立
→ 再支持 file-backed workflow
→ 最后考虑审阅、版本和复用策略
```

也就是说，持久化不是动态工作流的第一性要求。第一性要求是：任务程序能够被执行、被观察、被约束。

Claude Code 的公开文档也把 workflow script 作为可查看、可保存和可复用的资产来描述：运行过的 workflow 可以打开、修改、保存成命令，再在后续任务里用参数重新运行。这说明 file-backed workflow 是一个自然的演进方向。但它应当建立在运行边界已经清楚之后，而不是一开始就把临时编排做成一个脚本管理系统。

这里还需要区分两种状态：

| 对象 | 保存的是什么 | 适合怎么处理 | 不应混入什么 |
| --- | --- | --- | --- |
| workflow source | 脚本本身、参数入口、依赖的 Agent / 工具约束 | 保存、diff、审阅、修改、复用 | 某次运行的中间结果 |
| workflow execution | 某次运行的阶段、已完成 agent、暂停点、临时结果 | 观测、暂停、恢复、终止 | 长期复用的流程定义 |

保存脚本不等于保存执行状态。Claude Code 的文档也把 resume 限定在同一 session 内：已完成 agent 可以复用结果，剩下的继续跑；退出后再启动则是新的运行。这是合理边界。脚本是资产，执行状态是运行时事实，不应该因为都叫 workflow 就混在一起。

saved workflow 还需要参数入口。Claude Code 用 `args` 让保存后的脚本接收结构化输入。这个设计点值得保留：如果一个 workflow 需要靠用户编辑源码才能复用，它就不是好资产；如果参数抽象过度复杂，它又会变成另一个配置系统。`args` 的价值在于给复用留一个窄入口。

## 什么时候不应该用 Dynamic Workflow

Dynamic Workflow 不是替代所有 Agent 编排。

如果流程稳定、强约束、长期复用，应该写固定 workflow。不要让模型每次重新生成审批流、账单校验或标准发布流程。

如果问题只是工具之间的数据搬运，代码化工具编排就够了。它不需要临时子 Agent，也不需要任务级角色。

如果任务只是一次问答，普通 Agent 更简单。动态工作流会增加额外运行成本和失败面。

如果任务必须严格经过人工审批或确定性规则，也不应该把核心控制权交给临时生成的程序。模型可以辅助生成计划，但放行条件必须由外部规则掌握。

如果任务成本不可控，也不应该直接放大到全量。动态工作流天然会产生更多子 Agent 和更多 token 消耗。公开文档建议先在小范围任务上试跑，这不是产品提示，而是工程原则：长任务需要预算、并发上限、总 agent 上限和可中止机制。

可以用一个判断来收束：

```text
任务路径需要根据现场证据变化
并且包含临时角色、并发、循环或阶段状态
并且这些控制结构值得从模型上下文里拿出来
```

同时满足这些条件，Dynamic Workflow 才开始有意义。

## 最后的边界

Dynamic Workflow 的价值不是“模型会写代码”。这句话太宽，也太容易把问题带偏。

真正的价值是：复杂任务里的临时控制逻辑，有了一个比 prompt 更明确、比固定 workflow 更灵活的位置。

模型负责生成适合当前任务的程序；运行时负责执行确定的控制结构；子 Agent 和工具仍然留在宿主框架的能力边界内；事件流继续把过程交给上层系统消费；最终结果再回到根 Agent 解释给用户。

这条边界成立时，动态工作流才不是一个黑盒脚本工具，而是一种任务级编排形态：

```text
ReAct 负责即时判断
固定 workflow 负责稳定流程
Dynamic Workflow 负责当前任务专属的可执行任务程序
```

复杂 Agent 系统最终要回答的不是“用了多少 Agent”，也不是“有没有代码”。而是那段真正推进任务的程序在哪里，谁生成它，谁执行它，谁观察它，以及谁有权让它结束。
