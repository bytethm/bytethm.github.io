一个 coding Agent 做不完任务时，最常见的反应是增加角色：先分析，再实现，再测试，再 review；review 不通过，就回到实现。

开始时这很有效。问题很快会出现：这次该不该先勘探代码？review 的意见应该回给哪个执行单元？测试失败是重试、回退，还是先补充信息？所有检查都过了以后，谁有资格说任务已经结束？

这些问题不能靠再加一个 Agent 消失。它们要求系统具备一段负责推进任务的逻辑。本文把这段逻辑称为**任务程序**：它保存任务状态，选择下一步，消费结果，并判断什么证据足以关闭任务。

这篇文章不比较组件清单。Graph、Team、Agent Tool、Remote Agent 都只是任务程序里的不同部件。真正值得区分的是：任务程序在哪里，由谁生成，由谁运行，以及谁拥有结束权。

## 多 Agent 为什么很快变成编排问题

“分析 -> 实现 -> review -> 不通过则回退”不是一组角色，而是一段控制逻辑。它已经包含了状态、顺序、回退边和结束条件：

```text
while !closure(state, evidence):
  action = next_action(state)
  result = execute(action)
  state, evidence = reduce(state, evidence, result)
```

缺少这段逻辑时，多个 Agent 只是一组可调用的执行单元。它们可以各自完成任务，却没有人决定下一个该做什么，也没有人能区分“模型暂时停了”和“任务确实完成了”。

这也是许多 multi-agent 方案的误区。系统遇到复杂任务后加入更多 Agent，最后却把状态、路由和验收又塞回一个 manager prompt。执行资源增加了，任务程序仍然隐含在模型上下文里。

任务程序不一定要写成代码。它也可以藏在一轮轮模型推理中。但一旦任务需要回退、并发、重试、验收或恢复，就必须先问：**这段程序现在在哪里？**

## 第一种答案：人先把过程写出来

固定 workflow 将任务程序放进代码或 Graph。哪些阶段必须经过、什么可以并发、失败回到哪里、什么条件下放行，都在运行前确定。

这不是“传统方案不够智能”。它解决了一个具体压力：已知过程不应在每次运行时重新被模型发明。实现 Agent 不能跳过 review；验收不通过也不会因为一句自然语言收尾而被当作成功。

代价也很明确。固定流程只覆盖设计者预见到的路径。任务如果需要先调查、补充输入、改变策略，或者根本不需要完整流程，图就会开始膨胀。不断往 Graph 上加分支，并没有让它变得更灵活，只是把适用边界藏进更大的图里。

固定 workflow 适合的不是“简单任务”，而是**过程已经足够确定、值得被复用的任务**。

## 第二种答案：模型边想边决定

普通 ReAct 采取相反的选择：任务程序不单独存在，而是留在模型的上下文和下一轮推理里。

```text
state -> model chooses next action -> execute -> new state
```

它解决的是固定流程的盲区。模型可以发现未知项后先检索，可以在测试失败后换一种实现路径，也可以根据当前证据决定是否值得继续。这种现场判断不能被低估。

但 ReAct 的代价也在这里。计划、中间结果、路由判断和完成判断混在同一段上下文中。系统无法单独检查“为什么这一步被跳过”，也无法保证模型不会在还缺验收证据时自然收尾。

因此，ReAct 不是没有任务程序；它是把任务程序作为一次次即时推理来执行。灵活，但难以复用、审计和约束。

## 第三种答案：模型生成过程，运行时执行过程

固定 workflow 与 ReAct 之间还有第三种形态：模型先为当前任务生成一段专属编排，再交给运行时执行。

它的关键不在于“模型会写 JavaScript”。普通 Agent 当然也能写脚本。关键在于脚本被提升为一个运行时对象：运行时保存阶段和中间结果，调度受控 Agent，限制并发与资源，展示进度，处理暂停和恢复，最后回收结果。

Claude Code 的 Dynamic Workflow 是这类设计的例子：模型针对任务生成 JavaScript 编排脚本，隔离 runtime 调度子 Agent；脚本可被审阅、保存与重跑。[官方文档](https://code.claude.com/docs/en/workflows)

![三种任务程序的形态：固定流程图、ReAct 循环和动态生成的程序交给运行时调度多个 Agent。](/assets/task-program-forms.png)

*图 1. 固定 Workflow 由人预先定义；ReAct 由模型每轮解释；Dynamic Workflow 则由模型生成一次任务专属程序，再由运行时推进。Agent 数量不是这三者的本质区别。*

把三种形态放在一起看，问题会更清楚：

| 形态 | 谁生成任务程序 | 谁决定下一步 | 程序保存在哪里 |
| --- | --- | --- | --- |
| 固定 Workflow | 人 | 代码 / Graph runtime | 定义文件、代码或图 |
| ReAct | 模型，每一轮 | 模型，每一轮 | 模型上下文 |
| Dynamic Workflow | 模型，每个任务一次或少数几次 | 脚本 / workflow runtime | 任务专属程序 |

这条路径否定了一个常见误解：灵活性不等于把调度权永久交给模型。模型可以负责生成编排；一旦编排成立，那些已经确定的循环、分支和并发关系应由运行时接管。

## 为什么任务程序需要运行时

“让 Agent 写一个 shell 脚本跑起来”看似已经实现了动态 workflow，实际上只完成了最小的一步：产生了一段代码。

一个任务运行时还要解决更多问题：哪些 Agent 可以被脚本调度，结果以什么结构返回，运行到哪一个阶段，哪些子任务已经完成，暂停后怎样继续，超出并发或成本边界时如何停止。没有这些边界，脚本只是一次工具调用；任务状态仍然散落在终端输出、模型上下文和人的记忆里。

![任务程序的分层：任务先形成计划，运行时执行阶段与并发 Agent，产生可消费的证据，再由独立的 closure 决定是否结束。](/assets/task-program-layers.png)

*图 2. 规划、运行、执行和结束不是同一层职责。规划可以由人或模型完成；运行时负责推进；结束必须以证据为依据。*

这个分层的价值不在于多一层抽象，而在于把不同的不确定性放回合适的位置：

- **规划层**决定任务程序从何而来：人预先设计、模型逐轮选择，或模型生成专属程序。
- **运行层**管理状态、分支、并发、重试、子 Agent 生命周期和资源边界。
- **执行层**完成具体工作：Agent、工具、Remote Agent、Graph 节点都在这里。
- **证据层**沉淀测试结果、结构化 finding、验证产物和明确的 blocker。
- **结束层**判断这些证据是否足以关闭任务。

把这几层混在一个 manager Agent 里，短期看起来更灵活，长期会让每一次失败都变成 prompt 调整问题。把它们全部固定在一个大 Graph 里，短期看起来更可控，长期又会让任何例外都变成流程改造。任务程序的价值，是让这两类约束可以分别落位。

## 推进任务，不等于允许结束任务

任务程序显式化以后，仍有一个最容易被忽略的边界：`next_action` 与 `closure` 不是同一件事。

```text
next_action = policy(state)
can_stop    = closure(state, evidence)
```

前者可以由模型、脚本或 Graph 计算。模型最适合处理依赖现场证据的选择：此刻是先调查、实现、review，还是询问用户。后者应当尽量依赖可验证的证据：测试是否通过、关键 finding 是否关闭、是否明确声明了 blocker。

[Goal 机制](../goal-mechanism/)改变的正是默认结束规则。普通 ReAct 的隐含规则是“模型不再调用工具，就结束”；Goal、todo contract、evaluator、verifier 则要求出现明确的终止状态或满足验收谓词：

```text
can_stop = all_todos_completed || blocker_declared
can_stop = evaluator(condition, transcript)
can_stop = tests_passed && verifier_passed
```

Goal 不负责选择下一步，也不替代 workflow。它只处理一个更保守的问题：没有足够证据时，系统不能把自然语言收尾当作任务完成。

## 不要按组件名选型

Graph、Team、Agent Tool、Remote Agent、Goal 并不在同一层：

- **Graph / Workflow** 可以承载整个任务程序，也可以只是某项宏能力的内部过程。
- **Team / 多 Agent** 描述谁参与执行与协作关系，不自动提供推进或关闭规则。
- **ReAct manager** 可以选择或生成下一项宏能力，但不应默认拥有结束权。
- **Goal / evaluator / verifier** 定义任务如何取得关闭资格。

一个 Graph 既可以是整个任务的控制器，也可以只是“代码审查”能力的内部实现。关键不是组件名称，而是它是否拥有任务级的推进权、状态和结束权。

## 先定位压力，再选择组件

设计复杂 Agent 系统时，先不要问“要不要多 Agent”。先回答下面的问题：

1. 任务真正缺的是执行资源，还是一段可复用的推进逻辑？
2. 哪些路径已经足够确定，值得从模型上下文移到代码或 Graph？
3. 哪些选择必须保留给模型，根据现场证据动态决定？
4. 执行单元返回的是聊天文本，还是可进入验收的结构化 evidence？
5. 谁计算 `closure`，它凭什么允许任务结束？

这些问题的答案不会导向一种固定架构。它们只会让边界变得诚实：模型负责它真正擅长的判断，运行时负责可重复的控制，验证器负责结束条件。

多 Agent 的意义也随之改变。它不再是给一个 Agent 加几个帮手，而是在构造、执行和约束一段任务程序。
